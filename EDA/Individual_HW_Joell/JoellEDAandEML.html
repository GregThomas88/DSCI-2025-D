# Add country data to routes
srcAirportFormat = airportRawData.set_axis(["SRC_AIRPT_ID","SRC_NAME","SRC_CTY","SRC_CTRY","SRC_IATA","SRC_ICAO","SRC_LAT","SRC_LONG","SRC_ALT","SRC_TZ","SRC_DST","SRC_DBTZ","SRC_TYPE","SRC_SRC"], axis=1)
destnAirportFormat = airportRawData.set_axis(["DESTN_AIRPT_ID","DESTN_NAME","DESTN_CTY","DESTN_CTRY","DESTN_IATA","DESTN_ICAO","DESTN_LAT","DESTN_LONG","DESTN_ALT","DESTN_TZ","DESTN_DST","DESTN_DBTZ","DESTN_TYPE","DESTN_SRC"], axis=1)

# Merge data that matches the IATA code
routeCountryIATA = pd.merge(routeRawData, srcAirportFormat, how= 'left', left_on= 'SRC_AIRPT', right_on= "SRC_IATA")
routeCountryIATA = pd.merge(routeCountryIATA, destnAirportFormat, how= 'left', left_on= 'DESTN_AIRPT', right_on= "DESTN_IATA")
routeCountryIATA = routeCountryIATA.loc[routeCountryIATA['AIRLINE'].notna()]

# Merge data that matches the ICAO code
routeCountryICAO = pd.merge(routeRawData, srcAirportFormat, how= 'left', left_on= 'SRC_AIRPT', right_on= "SRC_ICAO")
routeCountryICAO = pd.merge(routeCountryICAO, destnAirportFormat, how= 'left', left_on= 'DESTN_AIRPT', right_on= "DESTN_ICAO")
routeCountryICAO = routeCountryICAO.loc[routeCountryICAO['AIRLINE'].notna()]

# Final merge to create one data set
routeLocation = pd.concat([routeCountryIATA, routeCountryICAO])

Section 15: Compare and Contrasting Summary¶

First thing that I noticed is that the two models, given the same data provide differing quantities of "Optimal" results, even within different runs with the same model and data. After analyzing the dendrogram and deciding to go forward with 22 clusters, Hierarchical clustering always returns 25 results in cluster that is shared with the optimal direct route, that being just the Chicago to San Francisco route. For the Gaussian Mixture Model Clustering however, only around 9 to 16 are returned each run. Given that the Gaussian Mixture Model Clustering gives less results for the same amount of clusters, it may be safe to assume that this model might work better for us. Looking into all of the optimal replacement routes, we can see that there are many suggested routes that ended up in the optimal cluster in both methods, but there are some that were only added by one and not the other. A next step could be to set the resulting data up in a way that it can be easily plotted and routes visualized so that visual outliers could be noticed. The addition of the idea of a route versatility score should also be vetted, as it is something that I made up (with reasoning, however) in order to have more features to work with than just distance, but there could be better ways to rate a route, or find other more useful features to consider.

Overall, I think that these clustering methods have high potential to be useful in the grand scheme of our project with the goal of being able to recommend good alternatives to an originally planned route. I think once the team is able to work together on data engineering, there will be a lot more features that we'll be able to use for making even better cluster parameters, such as route timing.
